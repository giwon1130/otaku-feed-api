## 패키지 구조

`com.giwon.otakufeed`:

### `bootstrap/` — Spring 셋업
- `OtakuFeedApplication` — `@SpringBootApplication` 진입점
- `HealthController` — `/health` (인증 X)
- `JdbcConfig` — HikariCP DataSource + JdbcTemplate Bean
- `SecurityConfig` — JWT 필터 등록 + 보호된 path 설정
- `WebCorsConfig` — CORS 정책

### `features/` — 도메인별 vertical slice
각 feature는 `presentation/` (Controller) + `application/` (Service/Repository/Domain) 구조.

#### `auth/`
- `AuthController` — `/auth/*` endpoint
- `AuthService` — 로그인/회원가입 로직
- `JwtProvider` — JWT 생성/파싱
- `UserDomain` — User entity
- `UserRepository` — `otaku_users` 테이블 CRUD

#### `anime/`
- `AnimeController` — `/swipes`, `/swipes/bulk`, `/prefs`
- `SwipeRepository` — `otaku_user_swipes` 테이블 (upsert / bulkUpsert / find / delete)
- `PrefsRepository` — `otaku_user_prefs` 테이블
- `AnimeDomain` — `UserSwipe`, `SwipeResult` enum, `UserPrefs`

## 의존성 흐름

```
Controller → Repository → JdbcTemplate
   ↑           (no Service 중간계층 — 단순한 CRUD라 직결)
   │
JwtProvider (auth check 시)
```

**의도적 단순성**:
- Service 계층 거의 없음 (AuthService만 있음 — SSO 로직 때문)
- Repository는 JdbcTemplate 직접 (JPA 안 씀)
- Domain은 Kotlin data class (Entity ↔ DTO 분리 안 함)

근거: 백엔드는 인증 + 단순 sync. 비즈니스 로직 거의 없음 → 추상화 적게.

## 데이터 흐름

### 흐름 1: 로그인
```
POST /auth/login
  ↓
AuthController.login → AuthService.login
  ↓
UserRepository.findByEmail → BCrypt verify
  ↓
JwtProvider.generate
  ↓
{ token, userId, email, nickname } 응답
```

### 흐름 2: 좋아요 저장 (배치)
```
POST /swipes/bulk { swipes: [...] }
  ↓
extractUserId (JWT 파싱, 401 if invalid)
  ↓
SwipeRepository.bulkUpsert (batchUpdate, 1 RTT)
  ↓
{ saved: N } 응답 (gzip 압축)
```

### 흐름 3: 헬스체크
```
GET /health
  ↓
HealthController (인증 X)
  ↓
{ status: 'UP', application: 'otaku-feed-api' }
```

## 마이그레이션

- `src/main/resources/db/migration/V*__*.sql`
- Flyway가 부트 시 자동 실행
- 이미 적용된 마이그레이션은 스킵 (메타 테이블)
- schema는 `otaku` (default-schema)

현재:
- V1__init_schema.sql — users / user_prefs / user_swipes
- V2__add_oauth_columns.sql — OAuth 도입
- V3__optimize_swipe_indexes.sql — 정렬 인덱스 추가

## 부팅 동작

```
1. Spring 컨텍스트 시작 (lazy-initialization: true → 빈 deferred 생성)
2. HikariCP 풀 초기화 (minimum-idle: 1)
3. Flyway 마이그레이션 실행 (없으면 스킵)
4. 첫 요청 도착 시 관련 빈 lazy 생성
```

## 안 쓰기로 한 것

- ❌ JPA / Hibernate — JdbcTemplate으로 충분, 학습 곡선·복잡도 회피
- ❌ Spring Data Redis — 캐시 필요 정도 아님
- ❌ MapStruct — Domain 단순해서 수동 매핑
- ❌ Kotlin Coroutines — Spring MVC blocking이라 의미 없음
- ❌ GraphQL — REST로 충분
- ❌ R2DBC (reactive) — 동기 IO로 충분, 복잡도 회피

## 새 endpoint 추가 절차

### 1. Controller 추가
`features/<domain>/presentation/<Domain>Controller.kt`:

```kotlin
@PostMapping("/something")
fun doSomething(
    @RequestHeader("Authorization") authorization: String,
    @Valid @RequestBody req: SomethingRequest,
): ResponseEntity<SomethingResponse> {
    val userId = extractUserId(authorization)
    val result = repo.doSomething(userId, req.field)
    return ResponseEntity.ok(SomethingResponse(...))
}
```

### 2. Request/Response DTO
같은 파일 위에 `data class`:

```kotlin
data class SomethingRequest(
    @field:NotNull @field:Min(1) val field: Int,
)
data class SomethingResponse(val ...: ...)
```

### 3. Repository 메서드
`features/<domain>/application/<Domain>Repository.kt`:

```kotlin
fun doSomething(userId: UUID, field: Int): SomeRow {
    jdbc.update("INSERT INTO otaku_xxx ... VALUES (?, ?)", userId, field)
    return jdbc.query("SELECT * FROM otaku_xxx WHERE ...", rowMapper, userId, field).first()
}
```

### 4. (필요시) DB 마이그레이션
`src/main/resources/db/migration/V<N>__<설명>.sql`:

```sql
CREATE TABLE IF NOT EXISTS otaku_xxx (
    id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id   UUID NOT NULL REFERENCES otaku_users(id) ON DELETE CASCADE,
    ...
);

CREATE INDEX IF NOT EXISTS idx_xxx_user ON otaku_xxx (user_id);
```

⚠️ V 번호는 기존 최대 +1. 한 번 적용된 마이그레이션은 수정 X (운영 DB 깨짐).

### 5. (보호 필요시) SecurityConfig
- 기본은 `Authorization: Bearer <jwt>` 필수
- 익명 허용 path는 `SecurityConfig.kt`에서 명시 (예: `/health`, `/auth/**`)

## 핵심 패턴

### JWT 인증
```kotlin
private fun extractUserId(authorization: String): UUID {
    val token = authorization.removePrefix("Bearer ").trim()
    return jwt.extractUserId(token)
    // 잘못된 토큰이면 ResponseStatusException(401) 던짐
}
```

모든 보호된 endpoint 첫 줄에 호출. SecurityFilter가 무인증 차단까지만 함, userId는 controller가 직접.

### Repository 패턴
- `jdbc.update("SQL", args)` — INSERT/UPDATE/DELETE
- `jdbc.query("SQL", rowMapper, args)` — SELECT 다건
- `jdbc.queryForObject("SQL", rowMapper, args)` — SELECT 단건 (없으면 예외)
- `jdbc.batchUpdate("SQL", args[][])` — N개 입력 1 RTT

### upsert (PostgreSQL)
```kotlin
jdbc.update("""
    INSERT INTO otaku_user_swipes (user_id, anime_id, result)
    VALUES (?, ?, ?)
    ON CONFLICT (user_id, anime_id)
    DO UPDATE SET result = EXCLUDED.result, swiped_at = now()
""", userId, animeId, result.name)
```

### Validation
- `@Valid @RequestBody` + DTO에 `@field:NotNull` 등
- 실패 시 자동 400 + 메시지

### 에러 응답
- `runCatching { ... }.getOrElse { return badRequest() }` — 비즈니스 로직 실패
- `throw ResponseStatusException(HttpStatus.X, "msg")` — 명시적 HTTP 코드
- 처리 안 한 예외 → 500 (Spring 기본 핸들러)

## 안티 패턴

- ❌ Service 계층 무조건 만들기 — 비즈니스 로직 없으면 Controller → Repository 직결
- ❌ Entity ↔ DTO 분리 안 한 채 Domain 노출 — 응답 필드는 명시적 DTO로
- ❌ 마이그레이션 수정 (이미 적용된 것) — 새 V 추가
- ❌ `@Transactional` 남발 — 단일 쿼리 X. batchUpdate처럼 명시적 묶음일 때만
- ❌ 응답에 ID/timestamp 항상 포함 — 클라가 안 쓰면 빼서 egress 절감 (ADR-009 참고)

## 새 라이브러리 추가

`build.gradle.kts`의 `dependencies { ... }`에 추가:

```kotlin
implementation("group:artifact")        // 컴파일 + 런타임
runtimeOnly("group:artifact")           // 런타임만 (드라이버 등)
testImplementation("...")               // 테스트만
```

추가 후 `./gradlew build -x test`로 컴파일 확인.

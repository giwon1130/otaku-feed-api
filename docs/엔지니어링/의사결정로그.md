## ADR-007. JWT 파싱 실패 시 500 → 401

**시점**: 2026-04
**문제**: 잘못된/만료된 JWT 들어오면 `JwtProvider.extractUserId`가 파싱 예외를 그대로 throw → Spring 기본 핸들러가 500 Internal Server Error로 응답
**결정**: `runCatching { ... }.getOrElse { throw ResponseStatusException(401) }`로 감쌈
**근거**: 운영 가시성. Sentry/로그에서 진짜 500 vs 인증 실패 구분 가능

## ADR-006. HikariCP 도입 + lazy-initialization + V3 정렬 인덱스

**시점**: 2026-04
**문제**:
- `DriverManagerDataSource`는 풀링 안 함 → 매 요청 신규 connection
- `findByUserId ORDER BY swiped_at DESC`가 단순 인덱스로 sort 못 풀음
- 부팅 시간 길어 Railway cold start 회복 느림
**결정**:
- `HikariCP` (max 5, idle 1) — Railway 작은 인스턴스(512MB) 대응
- `spring.main.lazy-initialization: true`
- V3 마이그레이션: `(user_id, swiped_at DESC)` + `(user_id, result, swiped_at DESC)` 복합 인덱스
**측정**: health 응답 0.92s → 0.44s (2배 빠름)

## ADR-005. 비용 절감 1차 — gzip + bulk swipes + id 제거

**시점**: 2026-04
**문제**: Railway egress + 호출 수 비용
**결정**:
- `server.compression.enabled: true` (1KB 이상 본문 gzip)
- `POST /swipes/bulk` — `batchUpdate`로 N개 단일 RTT
- `SwipeResponse`에서 id 필드 제거 (UUID string ~40B/row × N rows 절감)
**근거**: 클라가 swipe debounce + bulk 패턴 도입했으니 짝 맞춰 백엔드도 bulk endpoint. id는 클라가 안 씀.

## ADR-004. Google/Kakao OAuth 도입

**시점**: 2026-04 이전
**문제**: 이메일/비밀번호만으론 가입 마찰
**결정**:
- `POST /auth/oauth/google` (Google ID token 검증)
- `POST /auth/oauth/kakao` (Kakao access token 검증)
- DB에 oauth_provider/oauth_subject 컬럼 추가 (V2 마이그레이션)
- password를 nullable로 (OAuth 사용자는 password 없음)
**비용**: Google Identity Services 라이브러리 + Kakao SDK 의존성

## ADR-003. JWT 30일 만료 + Refresh token 미도입

**시점**: 2026-04 이전
**결정**: `expiration-hours: 720` (30일). Refresh token 안 만듦.
**근거**: 친구/가족 위주 사용 가정. 30일은 사용자 마찰 적당. Refresh token 도입 시 보안 복잡도 증가 (refresh storage, rotation 정책 등)

## ADR-002. JdbcTemplate (JPA 미사용)

**시점**: 초기
**문제**: ORM 도입 vs 직접 SQL
**결정**: JdbcTemplate
**근거**:
- 비즈니스 로직 단순 (CRUD 위주)
- 학습 곡선 낮음
- SQL 최적화/EXPLAIN 직접 가능
- N+1 등 ORM 함정 회피

**비용**: row mapping 수동 (RowMapper 작성), 마이그레이션 수동

## ADR-001. 별도 백엔드 레포 (otaku-feed-api)

**시점**: 초기
**결정**: 클라(`otaku-feed`)와 분리된 별도 저장소
**근거**:
- 기술 스택 다름 (RN vs Spring)
- 배포 주기 다름 (앱은 빌드, 백엔드는 즉시)
- 권한 분리 (백엔드는 인증/secret 다룸)

**비용**: 변경 시 두 레포 PR 동시 (간헐적)

# 비용 최적화

> **누가 봄**: DevOps, BE
> **언제 봄**: 비용 알림, 분기 점검
> **함께 봐야 할 문서**: 클라 [`docs/운영/비용최적화.md`](https://github.com/giwon1130/otaku-feed/blob/main/docs/운영/비용최적화.md)

## Railway 과금 항목

| 항목 | 단위 | 비고 |
|------|-----|------|
| Compute (RAM × 시간) | GB-hour | always-on이면 24/7 비용 |
| Egress (outbound traffic) | GB | 응답 본문 크기 × 호출 수 |
| Storage (DB) | GB | PG add-on 분리 과금 |
| 무료 한도 | $5 credit/월 | Hobby plan |

## 적용된 절감 작업

### 응답 크기 (egress)
1. **gzip 압축** — JSON 응답 50–80% 감소 (`server.compression`)
2. **`SwipeResponse` id 제거** — UUID string ~40B/row × N rows 절감 (ADR-005)

### 호출 수 (compute)
3. **`POST /swipes/bulk`** — N개 swipe 단일 RTT (`batchUpdate`) (ADR-005)
4. **(클라 측) `apiHealth` keepalive 조건부** — 비로그인은 호출 0
5. **(클라 측) `apiMe` 1h 캐시** — 부팅마다 호출 안 함
6. **(클라 측) prefs/swipes 5분 TTL** — 자주 호출돼도 서버 안 때림
7. **(클라 측) swipe 3s debounce + bulk** — 빠른 5 swipe = 1 RPC

### Compute 효율
8. **HikariCP** — `DriverManager` → 풀링으로 connection 재사용 (ADR-006)
9. **`spring.main.lazy-initialization: true`** — 부트 시간 단축 (cold start 회복)
10. **DB 정렬 인덱스** — `(user_id, swiped_at DESC)` 등 (ADR-006) — sort 비용 일정

## 비용 측정 효과 (예상)

| 시나리오 | 작업 전 | 현재 |
|---|---|---|
| 비로그인 사용자 1시간 활성 | 121 호출 | **0 호출** |
| 로그인 사용자 1시간 (캐시 hit) | 121 호출 | 60 호출 |
| 빠른 swipe 20번 | 20 RPC | **1 RPC** (bulk) |
| `/swipes` 응답 (100 row) | full + raw | **gzip + id 제거 = ~30%** |
| Cold start 응답 시간 | 0.92s | **0.44s** |

## 모니터링

### Railway 대시보드
- **Metrics** → CPU/RAM/Network/Disk 그래프
- **Usage** → 월간 사용량 + 비용 추세
- **Notifications** → 사용량 임계 알림 (이메일)

### 권장 설정
- **Spending Limit**: $10/월 (사용자 직접 설정)
- **Notification 80%**: 사용량 80% 도달 시 이메일

## 추가 절감 옵션

### 즉시 가능
- **PgBouncer**: Railway add-on 도입 → connection pooling 재사용 (HikariCP 위에 추가 레이어)
- **Caffeine 캐시 (인메모리)**: `/prefs`, `/swipes` 사용자별 TTL 캐시 → DB 호출 절감
- **Postgres 인스턴스 다운사이징**: 사용량 작으면 작은 tier로

### 중장기
- **Cloudflare/CDN**: `/health` 같은 GET을 edge에서 응답 → Railway 호출 0
- **PostgreSQL EXPLAIN로 인덱스 효과 측정**: 실 데이터 쌓이면 V3 인덱스 효과 + 안 쓰는 단순 인덱스 drop
- **HTTP/2 keepalive 명시**: Railway가 처리 (확인 필요)
- **비동기 background job**: swipe write를 큐로 빼서 응답 빠르게 (현재 동기, 사용자 영향 없음)

## 비용 vs 시간 결정 기준

- **무료 한도 50%**: 모니터링 강화
- **80%**: 절감 작업 시작
- **100%**: 유료 전환 또는 대체

ROI 계산:
- 작업 시간 시급 환산 (예: $30/시간)
- 절감 작업 1시간으로 월 $1 줄어들면 30개월 ROI = 매력 없음
- 점증 비용 막거나 50% 이상 줄이는 작업만 우선
